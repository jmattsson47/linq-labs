The Big Picture

What "calling the database by services" actually means at Linq.

In The Codebase course you saw Paperclip: one repo, one server, one database. That's a monolith — and it's a great way to start. Linq started the same way. But as Linq grew, one codebase and one database became a single point of failure: if one part fell over, it could take the whole backend with it.

So Linq is doing what most companies do when they outgrow a monolith: peeling off pieces into independent microservices, each with its own database. Here's the shape of it today:

Analogy: The monolith is one big restaurant kitchen — one crew, one pantry. Wonderful when you're small. But if the pantry catches fire, no food comes out at all. Microservices are a fleet of food trucks — each with its own crew and its own pantry. One truck breaking down doesn't stop the others. The trade-off: the trucks now need radios to coordinate.

The Monolith (rolodeq)

One Rails app, one shared database — still the heart of Linq.

rolodeq is Linq's original backend: a Ruby on Rails application. It handles auth, core business logic, and most of the product. Everything inside it shares one PostgreSQL database called rolodeq_production, with a read replica to spread out heavy reads.

rolodeq/config/database.yml
ConnectionWhat it is
primaryThe main read/write database — rolodeq_production
primary_replicaA read-only copy, for queries that don't need to write

One shared database is exactly what makes a monolith easy at first: any part of the code can read any table, joins are trivial, and there's one place to look. It's also exactly what makes it fragile at scale — everyone shares the same fridge, so one bad query or one overloaded table can slow down everything.

Analogy: rolodeq is a big house-share where everyone cooks from one shared fridge. Super convenient — until someone leaves the door open or hogs all the space, and now nobody can cook.

"Service" Means Two Things

The single most confusing word in Linq's backend — untangled.

When someone at Linq says "service," they could mean one of two completely different things. They look similar and share a name, but one is just a chunk of code and the other is an entire running program. Click each to see the difference:

A service object = a Ruby class inside the monolith

Inside rolodeq there are about 200 "service objects" in app/services/. Each one is just a Ruby class that holds some business logic — for example add_user_to_organization.rb. They are not separate programs. They all run in the same Rails process and read from the same shared database.

rolodeq/app/services/ (examples)
FileWhat it does
add_user_to_organization.rbAdds a user to an org
accept_organization_invite.rbHandles invite acceptance
authentication_service.rbLogin / identity logic

Think of these as well-organized drawers in one big desk. Tidy, but still one desk, one room.

A microservice = its own program, its own database

A microservice like contact-service is a whole separate application: its own repository, its own server process, and crucially its own database. It runs on its own, can be deployed on its own, and can fail on its own without dragging the rest down.

Standalone microservices (examples)

These are separate buildings, not drawers. Each has its own pantry and its own front door.

So "the contact service" might mean a Ruby class or a standalone microservice depending on who's talking. The tell: does it have its own database and run on its own? If yes, it's a microservice. If it's just a class sharing rolodeq's database, it's a service object.

The Microservices

The real programs Linq has peeled off the monolith. Click each one.

Each of these is an independent application with its own database. Click to see what it does and what it owns:

contact-service

Job: Manages contacts. Stack: TypeScript / Fastify. Owns: its own PostgreSQL database, plus OpenSearch for fast contact search.

When another service needs "who is +1 555…?", it asks contact-service — it never reaches into contact-service's database directly.

phone-number-service

Job: Manages phone numbers. Stack: TypeScript / NestJS with the Prisma ORM. Owns: its own PostgreSQL database.

Other services route messages by asking this service which number to use.

zero-service

Job: A "backend-for-frontend" for the Zero product — it composes data from several other services for the app. Stack: TypeScript / NestJS. Owns: its own PostgreSQL database.

You'll see zero-service star in the end-to-end flow in the last lesson.

synapse

Job: The ultra-low-latency messaging core (iMessage delivery). Stack: Go. Owns: message events live in NATS streams rather than a classic table — built for speed and volume.

synapse is also where the event "language" the other services listen to is defined.

clickhouse-service

Job: Analytics — counting and reporting over huge volumes of events. Stack: Go. Owns: a ClickHouse database, a totally different engine tuned for analytics, not day-to-day app data.

Different job, different kind of database. Not every service uses PostgreSQL.

Notice the pattern: different services pick different databases for different jobs — and none of them share. That's the idea we unpack next.

Database-per-Service

The real punchline — why each service owns its own database.

This is the heart of the whole thing. In the monolith, every service object shares one database. Each microservice instead has its own database, with its own connection string:

# Each microservice points at a DIFFERENT database # contact-service DATABASE_URL=postgresql://.../contact_service # phone-number-service DATABASE_URL=postgresql://.../phone_number_service # zero-service DATABASE_URL=postgresql://.../zero_service

One shared database

Easy joins, one place to look.

But: one slow query or one bad migration can hurt everything. The whole backend shares the blast radius.

Database-per-service

Each failure stays contained — if contact-service's database struggles, phone-number-service is untouched.

But: no cross-service joins, and data can be briefly out of sync.

That isolation is what delivers the resilience you'd expect from "microservices": a problem in one service's database can't directly knock over another's. The cost is real, though — services can no longer just JOIN across each other's tables, so they have to ask each other for data and accept that things sync up a moment later instead of instantly.

Analogy: Separate fridges in separate apartments. A fire in one kitchen doesn't spoil everyone else's food. But you can't just grab milk from your neighbor's fridge anymore — you have to text them and ask. That "texting" is the next lesson.

How Services Talk

No shared fridge means they have to pass messages. Three ways they do it.

Because services can't read each other's databases, they communicate over the network. Linq uses three main mechanisms — click each:

NATS — the internal event radio

When something happens, a service announces it on a NATS "subject" and any interested service can listen. The announcer doesn't know or care who's listening. Subjects have a tidy naming convention:

// A real NATS subject from synapse synapse.event.message.sent.v1 synapse.event.message.delivered.v1 synapse.event.message.read.v1

This is great for "fire and forget" facts: a message was sent. Anyone who cares (analytics, webhooks, risk scoring) reacts on their own.

Webhooks — telling the outside world

Webhooks are how Linq notifies external customers' systems that something happened. Linq sends an HTTP POST with a standard, signed payload:

{ "event_type": "message.received", "data": { "message": { "id": "msg-1" } } }

They're HMAC-signed so the receiver can trust they really came from Linq, and retried if delivery fails.

HTTP — asking a direct question

Sometimes a service needs an answer right now. Then it makes a direct HTTP request to another service and waits for the reply — like a phone call instead of a broadcast.

// zero-service asks contact-service: who are these numbers? POST /v1/contacts/lookup { "phones": ["+15550001111"] } // contact-service replies { "+15550001111": { "name": "John" } }

This is the "text your neighbor for milk" pattern from the last lesson.

A Real Flow, End to End

Loading a chat list — watch the services cooperate without sharing a database.

Let's trace one real request: the Zero app opens and needs to show your list of chats, with each person's name and photo. No single database has all of that — so the services collaborate:

1

The app asks zero-service

The Zero web app opens a connection to zero-service and says "get my chats."

2

zero-service asks Synapse for the chats

It calls Synapse's API: GET /v3/chats. Synapse owns the messaging data and returns the raw chats — but those only contain phone numbers, not names.

3

zero-service asks contact-service for the names

It takes the phone numbers and calls POST /v1/contacts/lookup. contact-service owns the contact data and returns names and photos.

4

zero-service stitches it together

It merges "chats from Synapse" + "names from contact-service" into one tidy response.

5

The app shows your chat list

You see chats with real names and photos — assembled from three services, each owning its own slice of data.

No service reached into another's database. zero-service didn't JOIN Synapse's tables with contact-service's tables — it asked each one and combined the answers. That's database-per-service in action.

What you learned:

Monolith (rolodeq) = one Rails app, one shared database
Service object = a class inside the monolith (shared DB)
Microservice = its own program, its own database
Database-per-service = isolation & resilience, at the cost of joins
NATS / webhooks / HTTP = how services talk when they can't share a fridge
